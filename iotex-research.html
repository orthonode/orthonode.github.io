// 00 — Abstract

On February 21, 2026, IoTeX — a DePIN-focused blockchain — suffered a private key compromise resulting in the unauthorized withdrawal of USDC, USDT, IOTX, and WBTC from its TokenSafe contract, and the subsequent minting of approximately $4M in CIOTX and $4.5M in CCS via its MinterPool contract. Total estimated losses range from $2M (IoTeX official) to $8.8M (independent on-chain analysis). On-chain investigators linked the attacker's funding wallet to the $49M Infini exploit of February 2025, suggesting an organized threat actor conducting systematic, long-horizon, multi-protocol infiltration. IoTeX co-founder Raullen Chai stated publicly that evidence indicates the attack was planned six to eighteen months in advance.

This paper presents a forensic reconstruction of the attack and evaluates it against two components of the Orthonode Infrastructure Labs security architecture. The first is the Stylus Hardware Anchor (SHA) — a deployed hardware identity primitive on Arbitrum Sepolia that cryptographically binds execution authority to manufacturer-burned silicon eFuse identifiers via a four-gate on-chain verification model (89+ verified transactions). The second is the Orthonode Assurance Platform (OAP) — a deterministic lifecycle integrity governance engine in active development, whose published structural model addresses the behavioral monitoring gap present during the attacker's extended dwell period.

Under the stated integration assumptions and threat model defined in Section 2, the analysis finds that a SHA-gated privileged execution model would be expected to prevent each stage of the described exploit at the contract verification layer. All claims are explicitly bounded: SHA v1.0 targets ESP32-S3 hardware; OAP claims are grounded in published structural model only, not empirical deployment data.

1.2 Root Cause

Root cause: Access control failure via private key compromise. Execution authority was granted on the basis of cryptographic key validity alone — with no secondary verification of the signing entity's hardware identity, firmware state, or behavioral legitimacy.

The attacker employed no smart contract vulnerability. No reentrancy, flash loan, or oracle manipulation was involved. The attack succeeded because possession of a valid signing key was, by itself, sufficient to invoke privileged contract functions. Every transaction processed as legitimate because every signature was cryptographically valid. The system had no mechanism to verify the entity behind the key.

IoTeX co-founder Raullen Chai stated publicly that evidence indicates six to eighteen months of planning. On-chain investigator Specter linked the attacker's wallet to the Infini exploit ($49M, February 2025) — in which a former contract developer retained administrative privileges and executed a delayed drain. Both attacks share the same signature: long dwell time, insider-level key access, privileged contract abuse, cross-chain laundering via THORChain.

3.2 Architecture

// LAYER_01 — SILICON
ESP32-S3 eFuse Registers

Manufacturer-burned Base MAC (6 bytes) + Chip Model (1 byte) → Keccak-256 (0x01 Ethereum padding) → 32-byte HW_ID. Permanent, non-writable post-fabrication. Physically absent in virtual machines and cloud infrastructure.

// LAYER_02 — MIDDLEWARE
Python SDK

Receipt generation, device authorization flows, monotonic counter management, Ethereum-compatible Keccak-256 computation, on-chain submission helpers.

// LAYER_03 — VERIFICATION
Stylus / Rust / WASM Contract

Four-gate deterministic verification in native WASM. Fail-closed on any gate failure. Gas economics viable at scale via WASM amortization.

// 04 — Forensic Analysis: Gate by Gate

All findings are conditional on assumptions A1–A8 and correct SHA integration across all privileged execution paths. Language is framed conditionally throughout — not as absolute guarantees.

4.1 The Model SHA Proposes

// IoTeX current model: Execution_Authority = f(valid_signature) // SHA-gated model: Execution_Authority = f(valid_signature AND authorized_hw_id // Gate 1 AND approved_fw_hash // Gate 2 AND counter > last_counter // Gate 3 AND digest_valid) // Gate 4

Under assumptions A1–A5, the attacker satisfies the first condition. Without physical possession of a registered device running approved firmware with valid counter state, the remaining four conditions cannot be satisfied through software means alone.

4.2 The Four Gates Applied

// GATE_01 — IDENTITY
authorized_nodes[hw_id] == true
authorized_nodes[hw_id]

The attacker's environment — a compromised server or cloud VM — contains no ESP32-S3 with manufacturer-burned eFuse identifiers matching any authorized_nodes entry. Any submitted HW_ID is fabricated software data. Under assumption A2 and A3, the identity check returns false, causing execution to revert before any state changes occur. Neither the TokenSafe withdrawal nor MinterPool minting would proceed to subsequent processing.

Expected: execution reverts at identity check. Drain and minting do not begin.
// GATE_02 — FIRMWARE GOVERNANCE
approved_firmware[fw_hash] == true
approved_firmware[fw_hash]

Evaluated only if Gate 1 passes — which it would not under the stated threat model. However, Gate 2 is independently significant for the long-horizon infiltration scenario. An insider who modifies operational software over six to eighteen months produces firmware that no longer matches the hash approved during initial enrollment. Any modification — however incremental — produces a different Keccak-256 firmware hash. Gate 2 rejects it at contract level.

Expected: modified or unauthorized firmware rejected. Relevant to the 6–18 month dwell period.
// GATE_03 — REPLAY PROTECTION
counter > counters[hw_id]
ReplayDetected()

Each receipt's monotonic uint64 counter must strictly exceed the last value stored on-chain for that HW_ID. Any captured receipt is single-use — replaying it triggers ReplayDetected(). The attacker cannot manufacture a valid counter sequence without having previously submitted legitimate receipts through Gates 1 and 2 as an authorized entity. No valid counter state exists to build from.

Expected: ReplayDetected() on any replayed or fabricated counter sequence.
// GATE_04 — DIGEST VERIFICATION
Keccak-256 reconstruction
DigestMismatch()

The contract reconstructs the 32-byte digest from all receipt fields and compares against the submitted value. Substitution attacks — inserting a valid HW_ID from a registered device into an otherwise fabricated receipt — alter composite field values and produce a different digest. Under assumption S2 (Keccak collision resistance), DigestMismatch() fires on any tampered or composite receipt. Validated against 10,000+ cross-layer test vectors.

Expected: DigestMismatch() on any tampered, substituted, or composite receipt.

// 05 — OAP: The Behavioral Integrity Layer

OAP Status: Phase 1 foundation complete. Seven development phases remaining. Not yet deployed. All claims in this section are grounded in OAP's published structural and governance model — not empirical deployment data. Proprietary parameters (threshold constants, escalation breakpoints, decay coefficients) are not disclosed. Structural documentation available to qualified partners on request: infrastructure@orthonode.xyz

SHA addresses the moment of execution — it prevents unauthorized identities from invoking privileged functions. The IoTeX attack had a second dimension SHA alone cannot fully address: a six-to-eighteen-month dwell period during which the threat actor was present within the system before the final execution phase. OAP's published structural model is designed to emit a continuous behavioral integrity signal across exactly this kind of extended operational timeline.

5.1 Formal Definition

OAP is a deterministic lifecycle integrity governance engine that emits a trust score T ∈ [0, 100] as a deterministic function of a device's complete behavioral history, physical motion compliance, identity continuity, and firmware state. OAP does not enforce economic policy and does not block execution directly. It produces the integrity signal; downstream protocols apply their own policy against that signal.

5.2 Published Trust Score Model

T(t) = T_baseline - SUM[ S(eᵢ) × decay(t - tᵢ) ] // all violation events eᵢ - P(catastrophic) // if any catastrophic event exists bounded to [0, T_max] Where: T_baseline = 100 at enrollment S(eᵢ) = severity score — non-linear, monotonically increasing decay(t - tᵢ) = penalty reduction for recoverable events over time P(catastrophic) = permanent downgrade term — does not decay T_max = 100 (ceiling, floored at 0)

The formulation is fully reproducible from the signed event record by any party with access to the data, once calibration parameters are published. Same inputs always produce same output.

5.3 Device State Machine

5.4 Applied to the IoTeX Timeline

Applying OAP's published structural model to the IoTeX behavioral timeline reveals multiple points at which the trust score would be expected to degrade under the stated model:

  • Long dwell period (6–18 months): behavioral patterns inconsistent with legitimate administrative operation — unusual access frequency, atypical call patterns, firmware state inconsistencies — would accumulate scar events under OAP's motion integrity model, progressively reducing T(t) before final execution.
  • Minting phase: $4M CIOTX and $4.5M CCS minted in rapid succession represents velocity with no precedent in any legitimate administrative baseline. Under OAP's published severity model, this would be expected to cross catastrophic event thresholds, applying P(catastrophic) and moving the device to COMPROMISED state.
  • THORChain bridging: a behavioral signature with no legitimate operational analog. Expected to trigger FROZEN state under OAP's identity and anomaly detection model.
  • Override governance: even if an insider attempted to administratively reset the degraded trust score, OAP's published model caps overrides at 2 per rolling 365-day window, requires cryptographic signing, and commits every override to the append-only log. No hidden resets. No silent forgiveness.
Precision note: the above applies OAP's published structural model to the described scenario. Without deployed calibration parameters, these outcomes are structural projections — not empirically validated results. The stated outcomes assume behavioral deviations would cross OAP's proprietary severity thresholds.

// 06 — Adversarial Counter-Analysis

A technically rigorous evaluation requires engaging directly with the strongest objections a critical reviewer would raise.

6.1 Attacker steals an authorized device

If the attacker physically obtains an SHA-registered ESP32-S3 enrolled as an authorized node and running approved firmware with valid counter state, they possess everything needed to pass all four gates. This is a real attack vector. SHA's response is scope-bounded: SHA addresses the remote key compromise attack vector — the IoTeX scenario. Physical device theft is a different threat class requiring physical security controls (tamper-evident hardware, secure storage, decommissioning protocols) that are the operator's responsibility. SHA + physical security, not SHA as standalone absolute defense.

6.2 The governance (owner) key is compromised

If the contract owner key used to call authorizeNode() and approveFirmware() is itself compromised, an attacker could enroll a malicious device and approve malicious firmware. SHA addresses this in two ways: first, compromising a second key (the governance key) in addition to the execution key is a substantially higher barrier than the single-key compromise exploited at IoTeX. Second, governance operations can themselves be gated behind SHA verification of authorized governance hardware — SHA is composable with itself. This is a documented extension path.

6.3 ESP32 targets are not blockchain validators

This is the most substantive objection and is stated directly. SHA v1.0 targets ESP32-S3 microcontrollers. IoTeX's compromised infrastructure was server-grade validators. Two integration paths exist: validators incorporate an ESP32-S3 as a hardware signing coprocessor (signing key lives on-device, SHA receipt produced before any privileged transaction is submitted), or the SHA architecture extends to server-grade hardware (TPMs, HSMs) using equivalent immutable identifier logic. The four-gate model, receipt format, and Stylus contract are hardware-agnostic — the silicon layer is the integration variable. Neither path existed in IoTeX's stack on February 21, 2026. SHA is an architectural model demonstrating what would have prevented this attack class, not a system IoTeX had available and chose not to use.

6.4 Why not just use multi-sig?

Multi-signature schemes reduce single-key blast radius but do not eliminate it. If a sufficient number of keys are compromised — as in a long-horizon, multi-insider infiltration — execution proceeds. Multi-sig does not verify that co-signing entities are physically legitimate hardware rather than a coordinated set of compromised remote machines. SHA is complementary: multi-sig gated by SHA requires both a signing threshold and verified silicon identity from each co-signer — addressing both the single-key compromise vector and the coordinated insider vector simultaneously.

// 09 — Conclusion

The February 2026 IoTeX exploit demonstrates a persistent and systematically exploited vulnerability class: execution authority derived from cryptographic key possession alone, without hardware-level verification of the signing entity's physical identity, firmware integrity, or behavioral legitimacy. This vulnerability class has accounted for the majority of high-value crypto losses over eighteen months and shows a pattern consistent with an organized, long-horizon threat actor operating across multiple protocols.

SHA addresses this vulnerability through a four-gate on-chain verification model that binds execution authority to manufacturer-burned silicon identifiers. Under the threat model and assumptions formally defined in Section 2, a SHA-gated privileged execution model would be expected to prevent the IoTeX attack at Gate 1 — identity verification — before any withdrawal or minting operation could proceed. Gates 2 through 4 provide defense-in-depth against firmware modification, receipt replay, and receipt fabrication respectively.

SHA does not claim to prevent all possible attack vectors. Physical device theft, governance key compromise, and supply-chain tampering are explicitly out-of-scope and require complementary controls. SHA is correctly understood as a necessary addition to — not replacement for — HSM key management, multi-sig schemes, and operational security practices.

OAP addresses the behavioral dimension: the extended dwell period during which a threat actor is present but has not yet executed. Under OAP's published structural model, sustained behavioral deviation over a six-to-eighteen-month period would be expected to progressively reduce the device's trust score, giving downstream systems a verifiable, deterministic basis to gate or escalate before the final exploitation phase.
