// WHAT_IS_NEXUS : SYSTEM_OVERVIEW

The Problem It Solves

DePIN hardware nodes connect to the open internet — which means they are exposed to replay attacks, ingress spoofing, identity ambiguity, and economic tampering. Most existing setups rely on ephemeral tunnels with no enforced identity resolution, no fail-closed behavior, and no deterministic guarantee that execution reflects a valid physical state.

Nexus replaces ephemeral, ad-hoc connectivity with a persistent, hardened ingress architecture. The "Moment of Truth" moves to the edge — before any state transition is committed.

The Verify-then-Execute Standard

Nexus enforces a strict execution ordering: verification always precedes execution, without exception. The Sentry Bridge resolves the identity and environmental context of every inbound request. Only verified requests reach the Brain. Unverified or ambiguous requests are dropped immediately — no partial execution, no graceful degradation, no fallback to an unverified path.

This is the fail-closed principle: data loss is preferable to reward leakage.

What Nexus Is Not

Nexus is not a blockchain protocol. It is not a smart contract system, a consensus layer, or an on-chain settlement mechanism. Phase 1.4.0 is entirely off-chain — a sovereign edge gateway that governs execution state locally. On-chain anchoring (peaq, IoTeX) and multi-chain settlement are Phase 2.0+ targets, contingent on Phase 1.4.0 architectural validation.

Current Scope (Phase 1.4.0)

Infrastructure hardening is the sole focus. The node is capable of sustained throughput (50–60 TPS, 1M transaction stress test, 0% corruption), fail-closed ingress is operational, and the deterministic economic invariant (60/30/10 split) is enforced at the protocol layer. Crypto-level hardware identity (peaq ID, ioID) is in progress and deferred to Phase 2.0.

LAYER_02

Cloudflare Zero Trust
(Hardened Ingress)

All hardware nodes communicate through encrypted Cloudflare Tunnels. Devices remain behind NAT with no public IPs and no open ports — eliminating the entire class of direct-exposure attacks present in public-internet deployments. Authentication is enforced at the Cloudflare Edge; only hardware with resolved identity context reaches the Nexus Controller.

The ingress layer is designed to be provider-agnostic. Cloudflare is the reference implementation. Operators may substitute Nginx, Traefik, or any compliant reverse proxy. Phase 2.0 targets sovereign P2P mesh (DHT peer discovery) to eliminate the remaining Cloudflare dependency.

Zero Exposure (No Public IP) DDoS Protection Identity-Gating at Edge Ingress-Agnostic Design
LAYER_03

Sentry Perimeter
(Trust Boundary)

The Sentry is the trust boundary of the Nexus node. Every inbound request is held at the Sentry until identity context is resolved. Requests carrying the correct Zero Trust routing headers pass to the Brain. Requests without valid context receive an immediate 403 — the Sentry never partially processes an unverified request. This is the fail-closed guarantee enforced at the perimeter, not as a downstream fallback.

Identity resolution in Phase 1.4.0 uses Cloudflare's header-based context. Cryptographic hardware identity (Sr25519 for peaq, Ed25519 for ioID) is the Phase 2.0 upgrade target.

Fail-Closed (403 on Identity Failure) Header Context Verification Crypto-Identity (Phase 2.0)
LAYER_04

The Brain
(Deterministic State Machine)

The Brain is a unidirectional state machine built for absolute persistence and auditability. It receives only verified, Sentry-cleared requests. All economic logic — the 60/30/10 split — is hardcoded here, enforced prior to every vault commit. No client-supplied parameters can override the split. Execution events are locally logged and replayable for deterministic audit.

Concurrency control uses Write-Ahead Logging (WAL) mode, validated under a 1-million transaction stress test: 0% corruption at 50–60 TPS sustained throughput on edge hardware. ACID compliance is verified. The Brain is the sovereignty layer — it owns the economic rules unconditionally.

Unidirectional State Machine 60/30/10 Invariant (Hardcoded) WAL Mode ACID Transactions 1M TX Stress Test Validated
LAYER_05

SQLite Vault
(Sovereign Ledger)

The Vault is the local sovereign ledger — a portable SQLite database operating in WAL mode. Every committed state transition is ACID-compliant and replayable. The vault file is portable: in a total hardware failure scenario, the .db file can be re-anchored to any Nexus-compliant node without state loss.

Vault encryption and backup strategy are operator-governed within the bounds of the Nexus governance model. Off-node anchoring (Merkle roots to peaq, IoTeX) is a Phase 2.0 target — the vault format is designed to be compatible with multi-chain state export.

SQLite WAL Mode ACID Compliance Portable (Hardware-Independent) Deterministic Audit Multi-Chain Anchoring (Phase 2.0)

// EXECUTION_FLOW : VERIFY_THEN_EXECUTE

STEP_01

Action Trigger

UI execution surface triggers an action (e.g. ledger split). Request exits only through the hardened ingress tunnel — no side-channel path exists.

STEP_02

Sentry Resolution

Sentry resolves identity context from routing headers. Request is held at the trust boundary — neither accepted nor forwarded until resolution completes.

STEP_03

Brain Execution

Verified request reaches the Brain. Deterministic logic runs. The 60/30/10 split is enforced unconditionally — no client parameter can override it.

STEP_04

Vault Commit

State transition is committed to the sovereign ledger under ACID guarantees. WAL mode prevents write collisions under concurrent access.

STEP_05

State Finalized

JSON confirmation returned to the execution surface. Event is logged locally for deterministic audit and replay.

VERIFIED PATH → Sentry clears context → Brain executes → Vault commits → State finalized
FAIL-CLOSED PATH → Identity ambiguity at Sentry → 403 Forbidden → Execution halted immediately → No partial state written

// THREAT_MODEL : PHASE_1.4.0_SURFACE

Scope clarification: Phase 1.4.0 threat model focuses on state correctness and ingress integrity. Insider threat (malicious node operator) is explicitly out of scope — the operator is the root of trust for their own hardware and vault. Network-level anonymity from state actors is not a Phase 1.4.0 objective.
T-01

Economic Tampering

Client attempts to alter the 60/30/10 economic split via injected request parameters or malformed payloads.

Mitigation: Split logic is enforced Brain-side only. The vault ignores any split parameters supplied by the client. The invariant is hardcoded and unreachable from the ingress path.
T-02

Ingress Spoofing

Attacker attempts to bypass the Cloudflare tunnel and hit the local controller port directly — circumventing the Sentry trust boundary entirely.

Mitigation: Sentry is configured fail-closed. Requests without valid Zero Trust routing headers are rejected immediately at the perimeter. Local port is not exposed.
T-03

Database Integrity Under Load

Concurrent writes, power loss, or hardware failure during vault commit corrupting the sovereign ledger.

Mitigation: SQLite WAL mode combined with atomic integer math. Validated under 1-million transaction stress test. 0% corruption observed at 50–60 TPS sustained throughput.
T-04

Ingress Provider Dependency

Cloudflare experiences an outage, applies rate limits, or applies censorship — severing the node's ingress path entirely.

Accepted risk in Phase 1.4.0. Architecture is ingress-agnostic by design. Operators may substitute Nginx or any compliant reverse proxy. Phase 2.0 targets sovereign P2P to eliminate this dependency.

Operator Authority

What the Operator Controls

Within the bounds of the Nexus invariant, the node operator has absolute autonomy over their deployment. They own the physical hardware, the OS environment, the ingress provider choice, vault encryption, and backup strategy. Once execution begins, however, the operator cannot retroactively modify committed state or alter transaction finality. The sovereignty of the ledger is unconditional.

Protocol Authority

What the Protocol Controls

The 60/30/10 economic split is a protocol-level invariant. It is hardcoded in the Brain, enforced before every vault commit, and structurally unreachable from the ingress path. Any attempt to bypass or modify it at runtime triggers an immediate fail-closed execution halt. The operator controls the machine — not the rules.

// HIVEMAPPER_BEE_RUNTIME : REFERENCE_IMPLEMENTATION

What It Is

The Hivemapper Bee Runtime is a hardware-specific Nexus sub-module targeting ARMv8 dashcam devices (Cortex-A53). It applies vibration entropy filtering at the edge layer — distinguishing genuine physical road data from stationary or emulated data streams before the payload reaches the ingress pipeline.

It is the first concrete instantiation of the Nexus Verify-then-Execute model on a specific DePIN hardware target, and serves as the reference for future hardware runtime implementations.

Relevance to Nexus

The Bee Runtime demonstrates that Nexus's modular adapter architecture works in practice against a real hardware profile — not just a reference simulation. Physics-based filtering (vibration entropy) maps directly to OAP's motion integrity doctrine, making the Bee Runtime a concrete bridge between Nexus's edge infrastructure layer and OAP's higher-level integrity governance model.

Benchmarks for the Bee Runtime are maintained separately within the repository.
