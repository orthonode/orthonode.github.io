THE_PROBLEM

DePIN Gateways Trust First, Verify Later

Most edge gateways accept transactions optimistically and validate after the fact — creating a window for spoofed telemetry, injected commands, and unauthorized execution to reach actuators, contracts, and reward pools before any check occurs.

Nexus Protocol inverts this model: nothing executes before verification completes.

THE_SOLUTION

Verify-Then-Execute. Fail-Closed.

Every transaction entering the Nexus gateway passes sequential identity, integrity, and authorization gates. Only verified transactions reach the execution layer. Unverified or malformed transactions are dropped at the gateway — they never reach downstream systems.

The Sentry module enforces fail-closed behavior: if the verification pipeline fails for any reason, the gateway closes rather than defaulting to pass-through.

TECH_STACK

Python / FastAPI · Internal Routing

Nexus Protocol is implemented in Python with FastAPI for the gateway API layer. Internal routing follows a 60/30/10 model — 60% primary path, 30% secondary verification path, 10% audit/telemetry path — ensuring verification coverage across all traffic patterns without single-path bottlenecks.

VALIDATION

1,000,000 TX Stress Test · 0% Corruption

Phase 1.4.0 completed a 1,000,000 transaction stress test at 50–60 TPS with 0% data corruption. The Fail-closed Sentry module is operational. All verification gates (identity, integrity, authorization) passed full stress coverage. Results published with Phase 1.4.0 release.

// INTERNAL_ROUTING_MODEL : 60/30/10

60%
Primary Verification Path

Full-Gate Transaction Processing

The primary path handles all standard transactions through the complete verification pipeline: identity gate → integrity gate → authorization gate → execution. Full SHA receipt validation, OAP integrity check, and authorization token verification on every transaction.

30%
Secondary Verification Path

Redundant Cross-Validation

The secondary path runs parallel verification against a secondary identity registry and authorization store. Discrepancies between primary and secondary verification results trigger immediate Sentry escalation. Prevents single-path compromise from bypassing the gateway.

10%
Audit & Telemetry Path

Full-Fidelity Transaction Log

A 10% sample of all transactions is captured to the audit path for full-fidelity telemetry logging, anomaly detection, and behavioral baseline analysis. Audit path data feeds OAP integrity scoring and is retained immutably for compliance purposes.

// SENTRY_MODULE : FAIL_CLOSED_ENFORCEMENT

// SENTRY_OPERATIONAL · PHASE_1.4.0

If Verification Fails, the Gateway Closes

The Sentry module monitors the verification pipeline in real time. If any gate in the primary or secondary verification path fails — hardware identity mismatch, integrity violation, authorization rejection, or pipeline fault — the Sentry closes the gateway immediately.

Fail-closed means transactions cannot reach the execution layer by exploiting a verification failure. There is no "degrade gracefully" mode. There is no fallback to pass-through. The Sentry enforces binary outcomes: verified and executed, or failed and closed.

Sentry escalation events are logged immutably to the audit path with full transaction context, gate failure reason, and timestamp. All escalations are reviewable and cross-referenced against OAP behavioral history for the originating device.

// FREQUENTLY_ASKED_QUESTIONS

What is the Nexus Protocol and what does it verify?

Nexus Protocol is a sovereign zero-trust edge gateway for DePIN hardware. It enforces verify-then-execute: every transaction passes identity, integrity, and authorization gates before any downstream execution occurs. Identity gates check device authorization (SHA receipt validation). Integrity gates check OAP behavioral trust signals. Authorization gates check execution permissions. Only transactions passing all gates are executed. All others are dropped at the gateway.

What does "fail-closed" mean in the context of the Sentry module?

Fail-closed means the gateway closes — not degrades, not passes through — if the verification pipeline fails. The Sentry module monitors every gate in real time. A pipeline fault, hardware identity mismatch, integrity violation, or authorization rejection causes Sentry to close the gateway immediately. Transactions cannot reach the execution layer by exploiting a verification failure. There is no fallback mode, no configurable "permissive" option, and no bypass path.

What was validated in the Phase 1.4.0 stress test?

Phase 1.4.0 completed a 1,000,000 transaction stress test at 50–60 transactions per second. The result was 0% data corruption across all verified transactions. The full 60/30/10 internal routing model was exercised: primary verification path (60%), secondary redundant path (30%), and audit/telemetry path (10%). The Fail-closed Sentry module was operational throughout and responded correctly to all injected fault conditions.

How does Nexus Protocol integrate with SHA and OAP?

Nexus Protocol consumes SHA and OAP as verification inputs. The identity gate validates SHA hardware receipts — confirming the originating device is authorized and its silicon identity is verified on-chain. The integrity gate checks OAP lifecycle trust signals — confirming the device's behavioral history meets the integrity threshold for the requested execution. Both integrations are planned for Phase 2.0 (SHA) with OAP signal integration already live in Phase 1.2+.
