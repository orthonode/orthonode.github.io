// WHAT_IS_SHA : PROTOCOL_OVERVIEW

The Problem It Solves

Arbitrum has no reusable, protocol-level primitive for binding physical device identity to on-chain execution. Every team building hardware-integrated applications must implement custom cryptographic verification logic from scratch — introducing fragmented security assumptions, inconsistent replay protection, and repeated implementation risk across the ecosystem.

SHA eliminates this duplication by providing a shared, auditable infrastructure layer that any Stylus contract can import.

What SHA Provides

SHA binds immutable silicon identifiers (ESP32 eFuse values burned at manufacture) to Stylus smart contracts using a gas-efficient, cryptographic verification protocol. It enforces four core guarantees — identity gating, firmware governance, replay protection, and cryptographic proof — within a single modular primitive.

Built in Rust/WASM via Arbitrum Stylus, SHA achieves lower gas overhead than equivalent Solidity implementations, making hardware verification economically viable at scale.

What SHA Is Not

SHA is not an oracle system, an enclave-based attestation service, a centralized hardware verification provider, or a single-protocol solution. It relies on no trusted intermediary — all verification logic executes deterministically on-chain inside the Stylus WASM runtime.

Current Status

A research prototype is live on Arbitrum Sepolia. Core feasibility has been validated: eFuse-based hardware binding works, Keccak-256 parity between device and contract has been confirmed across 10,000+ test vectors, and gas benchmarks have been published. Production hardening — fuzzing, threat modeling, audit preparation — is the next phase, pending grant approval.

// SYSTEM_ARCHITECTURE : THREE_LAYER_STACK

01

Silicon Layer — ESP32-S3

The hardware root of trust. Manufacturer-burned eFuse identifiers (Base MAC + Chip Model) are extracted at runtime and composed into a 32-byte Hardware Identity (HW_ID). On-device Keccak-256 hashing (Ethereum-compatible, 0x01 padding) produces receipt digests that match on-chain computation deterministically.

ESP32-S3 eFuse Base MAC + Chip Model Keccak-256 (0x01 pad) 32-byte HW_ID Monotonic Counter
02

Verification Layer — Stylus / Rust / WASM

The on-chain enforcement layer. Stylus contracts reconstruct receipt digests from submitted parameters and compare them against device-generated values. Monotonic counter enforcement prevents receipt replay. Firmware hash gating restricts execution to devices running approved firmware versions. All logic executes in the Stylus WASM runtime, achieving lower gas overhead than equivalent Solidity implementations.

Rust (Stylus SDK 0.6.x) WASM Execution Replay Protection Firmware Hash Gating Identity Registry
03

Middleware Layer — Python SDK

Developer-facing tooling for device authorization flows, receipt generation utilities, and on-chain submission helpers. The SDK exposes an async interface with configurable RPC endpoints, enabling multi-chain support across Arbitrum One, Nova, and Orbit chains by simply changing the provider URL. Published to PyPI as anchor-verifier.

Python async interface Configurable RPC PyPI: anchor-verifier Multi-chain support

// VERIFICATION_FLOW : DEVICE_TO_CONTRACT

STEP_01
Hardware Identity

ESP32-S3 reads manufacturer-burned eFuse values. Base MAC and Chip Model compose a 32-byte HW_ID unique to each physical device. This identity cannot be cloned or emulated in software.

STEP_02
Receipt Generation

The device generates a receipt: a Keccak-256 digest over (HW_ID ∥ firmware_hash ∥ counter ∥ payload). The monotonic counter increments on each submission, making receipts non-replayable by construction.

STEP_03
On-Chain Submission

The Python SDK submits the receipt parameters to the Stylus contract via EIP-1559 transaction. Single or batch submission paths are both supported.

STEP_04
Contract Verification

The Stylus contract reconstructs the digest, checks the identity registry, validates the firmware hash, enforces counter monotonicity, and emits a verified event. Failure reverts with a typed error.

// CORE_GUARANTEES : FOUR_PRIMITIVES

Identity Gate

Device-Bound Execution

Only transactions originating from pre-authorized hardware identities can execute gated contract functions. Silicon eFuse values are burned at manufacture and cannot be reprogrammed, making hardware identity immutable and Sybil-resistant.

Firmware Governance

Approved Firmware Enforcement

Contract execution is restricted to devices running firmware versions whose hashes have been explicitly approved by the contract owner. This creates a governance-controlled execution layer — outdated or compromised firmware is automatically rejected on-chain.

Replay Protection

Monotonic Counter Enforcement

Each hardware receipt embeds a monotonically increasing counter enforced by the Stylus contract. Submitted receipts must have a counter strictly greater than the last accepted value for that device. Replayed or duplicated receipts are deterministically rejected.

Cryptographic Proof

On-Chain Digest Validation

The Stylus contract independently reconstructs the Keccak-256 receipt digest from submitted parameters and compares it against the device-generated value. No trusted oracle or off-chain co-signer is required — verification is fully deterministic and self-contained on-chain.
