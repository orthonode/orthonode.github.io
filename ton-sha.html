THE_PROBLEM

TON Agents Have No Verifiable Identity

TON agents currently have no verifiable identity and no proof of execution integrity. Any agent can claim to be any other. Any output can be fabricated without cryptographic proof of the computation that produced it. There is no protocol-level primitive for agent trust on TON.

TON-SHA is the cryptographic primitive that changes that.

THE_SOLUTION

One Call — Trust or Reject

The canonical TON-SHA loop: one contract call tells any protocol whether the agent on the other side should be trusted. Four gates execute sequentially — identity, firmware, counter, digest. All four must pass. Any single failure reverts.

Composable with agent marketplaces, oracle networks, DeFi protocols, DAOs, and DePIN reward systems on TON.

// FOUR_GATE_VERIFIER : SEQUENTIAL_TRUST_CHECK

Gate 1 — Identity
Device Authorization

agent_id must be in the authorized registry. The agent_id is bound to the TON wallet key — cannot be forged or transferred without on-chain action.

Blocks: Sybil · Impersonation
Gate 2 — Firmware
Firmware Approval

firmware_hash must match a hash in the contract's approved firmware registry. Blocks agents running unauthorized or tampered software versions.

Blocks: Firmware Tamper
Gate 3 — Counter
Replay Protection

counter must be strictly greater than the last accepted counter for this agent. Monotonic enforcement — mathematically airtight replay prevention.

Blocks: Replay · Rollback
Gate 4 — Digest
SHA-256 Verification

SHA-256(agent_id ‖ firmware_hash ‖ execution_hash ‖ counter) == digest. Cannot be faked without running the actual computation that produced the execution_hash.

Blocks: Fabrication · Cache

// FREQUENTLY_ASKED_QUESTIONS

What is TON-SHA and who is it for?

TON-SHA is a cryptographic agent trust primitive for the TON blockchain. It provides four-gate verification — device authorization, firmware approval, replay protection, and SHA-256 digest validation — in a single Tact 1.6 contract call at 0.03 TON per operation. It is designed for TON agents (Telegram bots, automated trading agents, DePIN nodes) and any TON protocol that requires provable execution. No trusted intermediary is involved — all verification is deterministic and on-chain.

How does TON-SHA relate to SHA on Arbitrum?

TON-SHA is the TON ecosystem equivalent of SHA (Stylus Hardware Anchor) on Arbitrum. SHA uses Keccak-256 on Arbitrum Stylus (Rust/WASM) for hardware-bound identity. TON-SHA uses SHA-256 in Tact 1.6 on TON for agent identity and execution receipts. Both enforce four-gate verification: identity, firmware, replay, digest. In INVARIANT, SHA provides Layer 1 (hardware identity) and TON-SHA provides Layer 2 (execution receipts) of the trust stack.

What makes TON-SHA replay-safe?

Gate 3 enforces monotonic counter replay protection. Each agent maintains a counter that must be strictly greater than the last accepted value for that agent. The contract stores the last accepted counter per agent and rejects any submission where the counter is not strictly increasing. This makes it mathematically impossible to replay a previously accepted receipt — even if an attacker captures the exact bytes of a past valid submission, re-submitting it will fail Gate 3.
