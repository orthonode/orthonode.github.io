The Security Model

Four gates. No exceptions.

Every VerifyReceipt message passes through all four gates in sequence. Failure at any gate emits a specific on-chain failure code and aborts immediately. There is no partial success, no bypass path, no fallback.

01

Device Authorization

The contract owner explicitly authorizes each agent ID before it can submit any receipt. Unknown agents are rejected at first check â€” before any hash computation or storage access runs.

authorized_devices[hw_id] == true

Emits: VerificationFailed { hw_id, reason: 1 }

02

Firmware Approval

The SHA-256 hash of the agent's running firmware must be in the owner-curated approved set. An agent running modified, unrecognized, or injected code fails here regardless of its device identity.

approved_firmware[fw_hash] == true

Emits: VerificationFailed { hw_id, reason: 2 }

03

Replay Protection

Each receipt carries a monotonic counter that must strictly exceed the last accepted counter for that device. Captured receipts can never be replayed. Each successful verification permanently advances the counter on-chain.

counter > counters[hw_id]

Emits: VerificationFailed { hw_id, reason: 3 }

04

SHA-256 Integrity

The contract independently packs the submitted fields into a cell and computes sha256 â€” the same way the hardware does. If even one bit of any field was modified in transit, the digest won't match and this gate fires.

sha256(hw_idâ€–fw_hashâ€–ex_hashâ€–counter) == digest

Emits: VerificationFailed { hw_id, reason: 4 }

Receipt Format

640 bits. Exactly.

The receipt is a packed TON cell containing exactly 640 data bits. The contract rebuilds the identical cell before running Gate 4 â€” field order, bit widths, and encoding must match exactly. Any deviation produces a different digest and fails verification.

hw_id 64 bits uint64 Â· eFuse identifier
fw_hash 256 bits uint256 Â· firmware SHA-256
ex_hash 256 bits uint256 Â· execution SHA-256
counter 64 bits uint64 Â· monotonic, BE
digest 256 bits int257 Â· sha256(above, packed)

Engineering Notes

Problems solved building this.

TON-SHA required solving several non-obvious problems across Tact, TON's address system, and cross-layer digest computation. Documented so the next builder doesn't spend days on the same issues.

Tact 1.6 contract parameter syntax incompatibility
FIXED
Tact v1.6 introduced contract Foo(param: Type) parameter syntax but it's incompatible with declaring additional storage fields in the contract body. The compiler rejects any field declarations when parameters are used. Fix: revert to the classic init(owner: Address) pattern with explicit field declarations.
SHA-256 signed vs unsigned: Gate 4 failing for ~50% of counter values
FIXED
Tact's sha256(slice) returns Int â€” a signed 257-bit integer. When the hash high bit is set (value â‰¥ 2^255), TVM interprets it as negative. But msg.digest was declared Int as uint256 â€” always positive. They never matched for ~50% of all sha256 outputs, causing deterministic Gate 4 failures for specific counter values. Fix: declare digest as Int as int257 and normalize the off-chain digest with raw >= 2^255 ? raw - 2^256 : raw.
subbuffer() returning null for specific cell bit alignments
FIXED
cell.bits.subbuffer(0, totalBits) returns null for certain bit counts. Passing null to sha256 hashes an empty buffer silently â€” producing a completely wrong digest with no error. This caused Gate 4 failures that only appeared at specific counter values. Fix: explicit null check with a slice.loadUint fallback loop.
EQ vs kQ address format checksum mismatch
FIXED
TON addresses exist in two forms â€” EQ... (non-bounceable) and kQ... (bounceable). Both point to the same contract but have different checksums. Address.parse() validates strictly and throws Invalid checksum for any mismatch. Always copy addresses verbatim from deploy output. Added checkOwner.ts as a diagnostic to confirm the target contract before any value transactions.
Transaction deduplication â€” counter not incrementing
FIXED
TON deduplicates transactions with identical valid_until timestamps submitted in rapid succession. Two quick sends in the same second were collapsed into one on-chain. Fix: 1-second enforced delay between scenarios in the demo scripts, plus 15-second post-transaction wait before reading getters (TON ~5s block time).
1 << 256 overflow in Tact constant evaluator
FIXED
TVM Int max is 2^256 - 1. Using 1 << 256 or pow2(256) to normalize the sha256 output causes the Tact constant evaluator to reject it at compile time â€” even in non-constant expressions. Fix: change the message field type to Int as int257 (native signed Tact Int) and handle normalization entirely off-chain before sending. No large literal needed in contract code.

Security

Trust model and attack surface.

TON-SHA's security rests on SHA-256 preimage resistance and correct TON VM execution. The contract does not trust the caller beyond what the gates enforce.

Receipt Replay

Attacker captures a valid receipt and resubmits it to gain spurious verification credit.

Gate 3: counters[hw_id] must be strictly exceeded. Counter is updated atomically on success and is permanent.
Receipt Forgery

Attacker constructs a receipt for an agent they don't control and forges a valid digest.

Gate 4: requires sha256 preimage knowledge. Without knowing the exact hw_id + fw_hash + ex_hash + counter, no valid digest is computable.
Unauthorized Device Registration

Attacker calls AuthorizeDevice to register their own rogue agent identity.

AuthorizeDevice checks sender() == self.owner. Transaction reverts on any other caller.
Firmware Substitution

Agent runs modified or injected firmware but passes Gate 2 anyway.

Gate 2: the firmware SHA-256 hash must be in the owner-approved set. Unrecognized hashes fail unconditionally.
Counter Front-Running

Validator reorders a VerifyReceipt to advance the counter before the legitimate device can.

TON has no public mempool. Even if front-run succeeds, only the counter advances â€” no impersonation occurs. Device generates next counter to resume.
Owner Key Compromise

Owner private key is stolen, enabling rogue device authorization and firmware approval.

Residual risk. Recommended mitigation: use a multisig wallet as the contract owner. Planned for Phase 3.
